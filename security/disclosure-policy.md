# ChangeSpec Security Disclosure Policy

This policy applies to the ChangeSpec specification, the reference implementations at `spec/reference/{go,typescript,python}`, the ChangeSpec Foundation's operated services (the registry and the planned key transparency log), and the conformance test suite. It does not cover third-party implementations of the specification; those should have their own disclosure policies.

**Last updated:** 2026-04-16
**Effective:** On v1.0 public launch.

---

## What this policy covers

The ChangeSpec Foundation commits to receiving and responding to reports of security vulnerabilities in:

- The ChangeSpec specification itself (flaws that cause any conforming implementation to be insecure).
- The reference implementations shipped with the spec.
- The ChangeSpec-operated infrastructure: registry service, well-known aggregation, key transparency log, signing CA, documentation hosting.
- The conformance test suite (flaws that allow non-conforming implementations to pass).

In scope:

- Flaws in signature design, canonicalization, key distribution, or trust model.
- Denial-of-service attacks against reference implementations or infrastructure.
- Authentication, authorization, or access control issues in the registry.
- Vulnerabilities in dependencies shipped by the reference implementations that affect security.
- Cryptographic implementation errors in reference validators or verifiers.
- Any flaw that leads to spoofed events being accepted as authentic.

Out of scope:

- Vulnerabilities in third-party implementations of the spec (report to those implementers).
- Vulnerabilities in third-party ChangeSpec platforms or aggregators (report to those operators).
- Social engineering or physical security.
- Brute-force attacks against authenticated services that use strong rate limiting.
- Self-XSS in unauthenticated areas of the documentation site.
- Reports generated by automated scanners without validation.

## How to report

### Primary channel

Email: `security@changespec.com`

PGP-encrypted submissions preferred. The ChangeSpec security team's public PGP key fingerprint is published at:

- `https://changespec.com/.well-known/security.txt`
- `https://changespec.com/security/pgp-key.asc`

Include in your report:

1. A clear description of the vulnerability.
2. The component(s) affected (spec section, reference implementation name, infrastructure service).
3. Steps to reproduce, including minimal test cases where applicable.
4. Your assessment of severity and impact.
5. Any proposed mitigations or patches.
6. Your preferred attribution (name, pseudonym, or anonymous).
7. Any disclosure deadline you require.

### Secondary channels

- GitHub Security Advisory on the main repository (private, preferred for code-level issues).
- Contact via a Foundation board member listed at `https://changespec.com/governance`.

### What not to do

- Do NOT open a public GitHub issue describing the vulnerability.
- Do NOT post to the public discussion forum, Discord, or social media until an embargo period has ended.
- Do NOT exploit the vulnerability beyond what is needed to confirm it.
- Do NOT access, modify, or delete data belonging to other users or the Foundation.
- Do NOT perform testing that degrades service availability for others.

## Response SLA

The Foundation commits to the following response times:

| Stage | Target | Hard limit |
|---|---|---|
| Acknowledgement of report | 24 hours | 72 hours |
| Initial severity assessment | 3 business days | 7 business days |
| Mitigation plan communicated to reporter | 7 days for critical, 14 days for high, 30 days for medium/low | 45 days |
| Fix deployed (infrastructure) | 14 days for critical, 30 days for high, 60 days for medium | 90 days |
| Fix released (spec/reference impls) | 21 days for critical, 45 days for high, 90 days for medium | 120 days |
| Public disclosure | Coordinated with reporter, default 90 days from report | 180 days |

Severity levels follow CVSS 3.1:

- **Critical:** CVSS 9.0-10.0. Immediate action required. Examples: arbitrary code execution via forged events, full trust bootstrap bypass.
- **High:** CVSS 7.0-8.9. Action required within 48 hours. Examples: signature verification bypass, event spoofing for arbitrary vendors.
- **Medium:** CVSS 4.0-6.9. Action required within a sprint. Examples: denial of service, information disclosure.
- **Low:** CVSS 0.1-3.9. Best-effort remediation. Examples: minor information leaks in error messages.

## Embargo and coordinated disclosure

The default disclosure model is coordinated 90-day disclosure:

- Day 0: Vulnerability reported to `security@changespec.com`.
- Day 0-7: Triaged, severity assigned, acknowledgement to reporter.
- Day 7-60: Fix developed, tested, and merged to private branches.
- Day 60-80: Coordinated release preparation. Major downstream implementers may receive advance notice under a fixed-duration embargo (typically 7-14 days) to prepare patches.
- Day 80-90: Public release of the fix, publication of the security advisory, CVE assignment (if applicable), public acknowledgement of reporter.

Reporters MAY request:

- **Shorter timelines** if the vulnerability is actively exploited in the wild. The Foundation will expedite.
- **Longer timelines** if the fix requires architectural changes (for example, a spec-level change requiring a minor version release). Timelines longer than 180 days require reporter consent.
- **Anonymous disclosure** - name not published in the advisory.
- **Credit-only disclosure** - name published but no contact details.

The Foundation reserves the right to disclose earlier than the agreed timeline if:

- The vulnerability is already public.
- The reporter engages in harmful behaviour (extortion, unauthorized data access, public disclosure before the embargo).
- The vulnerability is being actively exploited and users are at immediate risk.

## CVE assignment

The ChangeSpec Foundation is a CVE Numbering Authority (CNA) for ChangeSpec-identified components once v1.0 launches. Until then, CVE assignment is requested through the CVE Program's root CNA.

Reference implementation vulnerabilities receive CVE assignment. Specification vulnerabilities that require a spec revision receive CVE assignment if they have an assessable CVSS score.

## Acknowledgements and Hall of Fame

Reporters who responsibly disclose vulnerabilities are credited in:

- The security advisory itself.
- A Hall of Fame page at `https://changespec.com/security/credits`.
- The CHANGELOG entry for the fix release.

Credit is opt-in. Reporters who request anonymity are listed as "Anonymous Researcher".

## Bug bounty

**The ChangeSpec Foundation does not operate a paid bug bounty program at this time.**

The Foundation may offer:

- Swag (t-shirts, stickers) for first-reporter credit on any vulnerability.
- Speaker invitations to ChangeSpec community events.
- Public recognition via the credits page.

Commercial vendors who deploy ChangeSpec at scale are encouraged to run their own bounty programs covering their ChangeSpec integrations.

The Foundation reserves the right to introduce a paid bounty program in the future. This policy section will be updated when that happens.

## Safe harbor

The Foundation commits to not pursue legal action against security researchers who:

- Report vulnerabilities in good faith through the channels described here.
- Follow the guidelines in "What not to do" above.
- Give reasonable time for the Foundation to respond before public disclosure.
- Do not access, modify, or delete data beyond what is necessary to demonstrate the vulnerability.
- Do not extort, threaten, or coerce the Foundation.

This safe harbor applies to:

- Civil claims under the US Computer Fraud and Abuse Act, the EU Directive 2013/40/EU, and equivalent laws in other jurisdictions to the extent permitted by law.
- Claims under the Foundation's terms of service.
- Claims under applicable copyright law (DMCA and equivalents) for good-faith reverse engineering of the Foundation's services.

Researchers should consult their own legal counsel for advice specific to their jurisdiction.

## Spec-level vulnerabilities: special process

Vulnerabilities in the ChangeSpec specification itself (as opposed to implementations) follow a modified process because they cannot be patched invisibly. The process:

1. Report is received and triaged as usual.
2. The Foundation convenes a specification review panel (spec editors + security team) to assess.
3. If the finding requires a spec change:
   - For a clarification (no semantic change): patch release (1.0.1).
   - For an optional-field addition or a mitigation: minor release (1.1).
   - For a breaking change: security-driven major release with expedited deprecation.
4. The Foundation publishes a Security Advisory describing the flaw, the fix, and the migration path.
5. Reference implementations are updated to reflect the spec revision.
6. Existing implementations in the wild are contacted via the implementers' mailing list to request they upgrade.
7. CVE is assigned to the specification version.

Spec-level vulnerabilities disclosed after v1.0 launch will be tracked publicly at `https://changespec.com/security/advisories`.

## Infrastructure vulnerabilities

Vulnerabilities in Foundation-operated infrastructure (registry, transparency log, etc.) are treated with standard web-service disclosure:

- Fixes deployed without requiring spec changes.
- Post-incident review published within 30 days for incidents that caused user impact.
- Status page updates during active incidents.
- No embargo longer than 30 days for infrastructure-only issues (no implementation coordination needed).

---

## Supply chain vulnerabilities

Vulnerabilities in dependencies of the reference implementations are handled as follows:

- If the vulnerability is exploitable through ChangeSpec's use of the dependency: treat as a reference-implementation vulnerability with the standard process.
- If the vulnerability is not exploitable through ChangeSpec's use: track, pin to a fixed version on next release, but do not issue a ChangeSpec-specific advisory.

## Questions

General security questions that do not involve undisclosed vulnerabilities can be asked on the public discussion forum or by emailing `security-questions@changespec.com` (separate from the reporting address).

Policy questions about this document should be directed to the Foundation's governance board.
