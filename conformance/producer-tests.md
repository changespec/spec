# Producer Conformance Requirements

This document specifies the conformance requirements for **producers**: vendors, platforms, or services that publish ChangeSpec 1.0 events to consumers.

Requirements use RFC 2119 terms (MUST, SHOULD, MAY). MUST failures block certification. SHOULD failures are reported as warnings. MAY requirements are best practices with no certification impact.

---

## Section 1: Schema Conformance

### P-MUST-1.1 - All events must validate

Every event published by the producer MUST validate against the ChangeSpec 1.0 JSON Schema (`spec/schema.json`).

**Test method:** Submit 20+ real events to the certification tooling. All must pass `Validate()` with no errors.

**Example of failure:** An event missing the `summary` field, or using `severity: "extreme"` which is not in the enum.

### P-MUST-1.2 - specversion must be "1.0"

The `specversion` field MUST be exactly the string `"1.0"`. The field MUST be present in every event.

**Example of failure:** `"specversion": "1"`, `"specversion": "v1.0"`, or omitting the field.

### P-MUST-1.3 - id must be globally unique per producer

The `id` field MUST be unique across all events published by the producer. If the same logical change is re-delivered (e.g., retry), it MUST carry the same `id` as the original delivery (idempotent ID).

Producers SHOULD use the format `cs_` followed by a ULID or UUIDv4.

**Test method:** The certification sample must contain no duplicate `id` values.

**Example of failure:** A producer that generates a new UUID per delivery attempt, producing two different `id` values for the same event.

### P-MUST-1.4 - category must be a defined enum value

The `category` field MUST be one of the nine defined values: `api_breaking`, `api_deprecation`, `security`, `data_handling`, `liability`, `pricing`, `tos`, `cosmetic`, `informational`.

### P-MUST-1.5 - severity must be a defined enum value

The `severity` field MUST be one of the five defined values: `critical`, `high`, `medium`, `low`, `informational`.

### P-MUST-1.6 - source_type must be set correctly

The `source_type` field MUST accurately reflect how the event was produced:

- `publisher_verified` - used only when the vendor is the publisher and pushed the event directly.
- `crawled` - used only when the event was generated by automated change detection.
- `community` - used only when the event was submitted by a third party.

A producer using a platform API to publish events about its own changes MUST use `publisher_verified`.

**Example of failure:** A vendor using `crawled` for events they pushed directly through the API.

### P-MUST-1.7 - URL fields must be valid absolute URIs

When `source_url` or `migration_url` are present, they MUST be valid absolute URIs per RFC 3986. HTTPS URLs are REQUIRED for `publisher_verified` events. HTTP is not recommended.

**Test method:** Certification tooling makes a HEAD request to each URL and expects a 2xx or 3xx response.

**Example of failure:** `"source_url": "docs/changelog"` (relative URL) or `"source_url": "http://internal.host/changelog"` (non-HTTPS on a publisher_verified event).

### P-MUST-1.8 - published_at must be a valid RFC 3339 datetime

The `published_at` field MUST be a valid RFC 3339 datetime. It MUST NOT be set to a future timestamp (events are published when they happen, not scheduled into the future via this field).

**Example of failure:** `"published_at": "April 10, 2026"` or a timestamp two years in the future.

### P-MUST-1.9 - title must not contain newlines

The `title` field MUST NOT contain newline (`\n`) or carriage return (`\r`) characters.

### P-MUST-1.10 - Extension fields must use ext: prefix

If a producer adds custom fields beyond the standard set, they MUST use the `ext:<namespace>.<fieldname>` format. Custom fields without the `ext:` prefix are not permitted and MUST NOT be used.

## Section 2: Semantic Correctness

### P-MUST-2.1 - confidence_score must be 1.0 for publisher_verified

Events with `source_type: publisher_verified` MUST set `confidence_score` to `1.0` when the field is present.

**Example of failure:** A vendor-pushed event with `"confidence_score": 0.85`.

### P-MUST-2.2 - Security events must use the security category

Events describing CVEs, known vulnerabilities, credential exposures, or active exploits MUST use `category: security`. The `informational` or `tos` categories MUST NOT be used for security events.

### P-MUST-2.3 - cve_id must match the CVE pattern

When a `cve_id` field is present, it MUST match the pattern `CVE-YYYY-NNNNN` where `YYYY` is a 4-digit year and `NNNNN` is a sequence of at least 4 digits.

### P-MUST-2.4 - cvss_score must be in [0.0, 10.0]

When `cvss_score` is present, it MUST be a number in the range [0.0, 10.0].

### P-MUST-2.5 - effective_date and sunset_date must be YYYY-MM-DD format

When `effective_date` or `sunset_date` are present, they MUST be RFC 3339 full-date strings in `YYYY-MM-DD` format. Datetime strings with time components are NOT permitted in these fields.

**Example of failure:** `"effective_date": "2026-05-01T00:00:00Z"` (datetime, not a date).

### P-MUST-2.6 - action_required must be true for breaking changes

Events with `category: api_breaking` and `severity: critical` or `severity: high` MUST set `action_required: true`.

**Rationale:** Consumers use `action_required` for routing. A high-severity breaking change that sets `action_required: false` will be silently missed by automated routing.

### P-MUST-2.7 - vendor_id must match the naming format

The `vendor_id` MUST match the pattern defined in spec Section 3.1. No-namespace slugs must be globally unique company or service names. Namespaced IDs must use a registered namespace prefix.

## Section 3: Signature Requirements (publisher_verified events)

### P-SHOULD-3.1 - Publisher-verified events should be signed

Events with `source_type: publisher_verified` SHOULD include a valid `signature` block. Unsigned publisher-verified events are permitted but consumers that require signature verification will reject them.

### P-MUST-3.2 - If signature is present, alg must be "ed25519"

When a `signature` object is present, the `alg` field MUST be `"ed25519"`. No other algorithm is defined in ChangeSpec 1.0.

### P-MUST-3.3 - signature.value must be base64url encoded

The `signature.value` field MUST be a base64url-encoded string with no padding characters. Only characters in the set `[A-Za-z0-9_-]` are permitted.

### P-MUST-3.4 - signed_fields must include all required fields

When signing, the `signed_fields` array MUST include at minimum: `id`, `vendor_id`, `category`, `severity`, `title`, `summary`, `published_at`. Additional fields MAY be included.

### P-SHOULD-3.5 - Public keys should be published at well-known URL

Vendors issuing signed events SHOULD publish their Ed25519 public keys at `https://{vendor-domain}/.well-known/changespec-keys.json` per spec Section 7.4.

## Section 4: Operational Requirements

### P-SHOULD-4.1 - Events should include source_url

Events SHOULD include a `source_url` pointing to the primary source document for the change. Events without `source_url` cannot be verified by consumers and provide no path to further detail.

### P-SHOULD-4.2 - Deprecation events should include sunset_date

Events with `category: api_deprecation` SHOULD include a `sunset_date` indicating when the deprecated feature will be removed. Deprecations without a sunset date cannot be prioritized by consumers.

### P-SHOULD-4.3 - Crawled events should include confidence_score

Events with `source_type: crawled` or `source_type: community` SHOULD set `confidence_score`. Values below `0.7` indicate uncertain classification and SHOULD trigger human review before publication.

### P-SHOULD-4.4 - Security events should include affected_versions and fixed_in_version

Events with `category: security` SHOULD set `affected_versions` to indicate which versions are vulnerable and `fixed_in_version` to indicate the patched version, when applicable.

### P-SHOULD-4.5 - Events should not be delivered more than 24 hours after detection

Producers SHOULD publish events within 24 hours of detecting or learning of a change. Stale events erode consumer trust and may miss actionable windows.

### P-MAY-4.6 - Producers may include migration_hint and migration_url

For breaking changes and deprecations, producers MAY include `migration_hint` (a short plain-text action) and `migration_url` (a link to a detailed migration guide). These fields are not required but significantly improve consumer experience.

### P-MAY-4.7 - Producers may include affected_systems

When a change affects specific named systems within the vendor's portfolio, producers MAY list them in `affected_systems`. This aids consumer filtering.

### P-MAY-4.8 - Producers may include recommended_reviewers

For events that require review by specific teams (legal, security, compliance), producers MAY set `recommended_reviewers` to guide routing in consumer systems.

## Section 5: Content Quality (Advisory)

These requirements are advisory. They affect the usefulness of events but are not tested mechanically.

### P-MAY-5.1 - Title should be human-readable and specific

The `title` should describe the specific change, not a category. "API updated" is a poor title. "confirm() now requires return_url parameter in all redirect flows" is a good title.

### P-MAY-5.2 - Summary should be actionable

The `summary` should answer: what changed, what is the impact, and what should the consumer do (if anything). It should not require the consumer to read the `source_url` to understand whether action is needed.

### P-MAY-5.3 - summary must be plain text

The `summary` MUST NOT contain markdown syntax, HTML tags, or other markup. Consumers render the summary in contexts where markup will appear as raw characters.

## Certification Submission Checklist

For Level 4 certification, prepare:

- [ ] A set of at least 20 real events published by the producer in the last 90 days.
- [ ] Documentation of the production process (how events are generated, reviewed, and published).
- [ ] Confirmation that all events pass the Level 1-3 test suite.
- [ ] List of any P-SHOULD requirements not met, with rationale.
- [ ] For signed events: the public key document at the well-known URL, or documentation of an alternative key distribution mechanism.
- [ ] Contact information for the team responsible for maintaining conformance.

Submit via `changespec.com/certify`.
